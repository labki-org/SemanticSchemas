## v1 Requirements

### REQ-001: Clickable Wiki Links for Page-Type Properties
**Priority:** Must Have
**Source:** User request, research validation
**Status:** [x] Complete

Properties with `Has_type::Page` must render their values as clickable wiki links in display templates. Links should use standard MediaWiki wikilink syntax `[[Target]]`.

**Acceptance Criteria:**
- [x] Page-type property values display as clickable links
- [x] Links navigate to the correct wiki page
- [x] Works with existing three-template system (dispatcher/semantic/display)

**Outcome:** Validated - Template:Property/Page renders clickable wikilinks, smart fallback selects it automatically for Page-type properties.

---

### REQ-002: Multi-Value Support
**Priority:** Must Have
**Source:** User request
**Status:** [x] Complete

Comma-separated property values must each render as individual clickable links.

**Example:**
- Input: `Has parent category, Has target namespace`
- Output: `[[Property:Has parent category]], [[Property:Has target namespace]]`

**Acceptance Criteria:**
- [x] Multiple values separated by commas are split correctly
- [x] Each value becomes a separate wiki link
- [x] Links are joined with comma-space separator in display

**Outcome:** Validated - #arraymap handles multi-value splitting, &#32; entity ensures proper spacing.

---

### REQ-003: Empty Value Handling
**Priority:** Must Have
**Source:** Research recommendation
**Status:** [x] Complete

Empty or missing property values must render nothing (no broken links, no placeholder text).

**Acceptance Criteria:**
- [x] Empty string values produce no output
- [x] Missing parameter produces no output
- [x] No "[[]]" broken links in output

**Outcome:** Validated - #if wrapper produces no output for empty values.

---

### REQ-004: Smart Template Fallback Logic
**Priority:** Must Have
**Source:** User selection
**Status:** [x] Complete

Template selection for property display must follow this hierarchy:
1. If `Has_template` is set -> Use that custom template
2. Else if `Has_type` is `Page` -> Use `Template:Property/Page`
3. Else -> Use `Template:Property/Default`

**Acceptance Criteria:**
- [x] PropertyModel.getRenderTemplate() implements this fallback logic
- [x] Custom Has_template overrides always take precedence
- [x] Page-type detection uses existing isPageType() method

**Outcome:** Validated - Three-tier fallback implemented in PropertyModel.php.

---

### REQ-005: Template:Property/Page Implementation
**Priority:** Must Have
**Source:** Research architecture recommendation
**Status:** [x] Complete

Create new display template for Page-type properties using #arraymap for multi-value handling.

**Template Content:**
```wikitext
<includeonly>{{#if:{{{value|}}}|{{#arraymap:{{{value|}}}|,|@@item@@|[[:@@item@@]]|,&#32;}}|}}</includeonly>
```

**Acceptance Criteria:**
- [x] Template handles single values correctly
- [x] Template handles comma-separated multiple values
- [x] Template handles empty values (produces nothing)
- [x] Uses `@@item@@` variable (not `x`) to avoid collision

**Outcome:** Validated - Template deployed in extension-config.json Layer 0 with all criteria met.

## Out of Scope

- **SMW Storage Changes** - SMW already stores data correctly with namespace prefixes
- **Custom Link Formatting Per-Property** - Use Has_template for custom cases
- **Non-Page Datatypes** - This work is specific to Page-type properties
- **JavaScript Link Enhancement** - Anti-feature per research
- **#ifexist Validation** - Performance killer, anti-feature per research
